# Аудит Marine Track — 2026-07-10

## 1. Объём и исходное состояние

Аудит выполнен по `main` на коммите `7e7d6b0` (`Document experimental wake speed in README`). В репозитории 287 коммитов, открытых issues и pull requests нет. Проверены исходный код, конфигурация, документация, история последних изменений, тесты и CI-контракт.

Важно разделять три уровня готовности:

1. **Инженерно реализовано** — код существует и покрыт локальными тестами.
2. **Экспериментально доступно** — эвристика выполняется, но не имеет калиброванной точности и не должна трактоваться как измерение.
3. **Научно подтверждено** — есть отложенная выборка, независимая разметка/AIS, метрики по стратифицированным условиям и оценка неопределённости.

Текущее состояние проекта в основном находится между уровнями 1 и 2. Уровень 3 ещё не достигнут.

## 2. Итоговый вывод

Marine Track уже является рабочим каркасом эксплуатационного конвейера:

```text
AOI/время → поиск STAC-сцен → выбор raster asset → AOI crop → land mask → локальный CFAR → кандидаты → Hough wake → AIS enrichment → GeoJSON/CSV/Parquet/PNG/report
```

Но это пока **не подтверждённый детектор судов и не измеритель скорости**. Корректное описание текущего результата:

> Геопривязанные яркие кандидаты на одном GeoTIFF/COG с эвристическим признаком линейного следа, внешним AIS-сопоставлением и экспериментальным proxy скорости.

Главный риск — документация и конфигурация местами описывают целевую систему как уже реализованную. До научной валидации нельзя выводить `confidence` как вероятность обнаружения судна, а `speed_knots` по wake wavelength — как надёжную скорость судна.

## 4. Критические замечания

### P0-1. Release gate сейчас красный

В изолированном окружении Python 3.12 с установкой `.[dev]` получено:

```text
77 passed, 4 failed
```

Падают:

- `tests/test_detector_metrics.py` — на синтетическом объекте `contrast_sigma == 0`, потому что фон имеет нулевое стандартное отклонение; текущая формула не определяет fallback для детерминированного фона.
- `tests/test_runtime_check_profiles.py::test_provider_profile_alias` — тест ожидает удалённый alias `none`.
- `tests/test_runtime_check_profiles.py::test_numeric_env_validation_catches_invalid_values` — тест ожидает старое сообщение `must be integer`, код пишет `must be numeric`.
- `tests/test_smoke_check.py` — тест вызывает устаревший аргумент `check_telegram`.

`ruff check src tests` также не проходит: два неотсортированных import-блока (`detection_pipeline.py`, `telegram_bot.py`). `compileall` проходит, `runtime_check.py` проходит для `MARINE_TRACK_PROVIDER_PROFILE=core` с тестовым Telegram token. До исправления этих несоответствий release gate нельзя считать закрытым.

### P0-2. CDSE STAC в коде использует устаревший контракт

`src/marine_track/data_sources/stac_provider.py` жёстко задаёт:

```text
https://catalogue.dataspace.copernicus.eu/stac
collections: SENTINEL-1 / SENTINEL-2
```

Актуальный CDSE STAC API использует:

```text
https://stac.dataspace.copernicus.eu/v1/
sentinel-1-grd
sentinel-2-l2a
```

Старый endpoint официально deprecated с 17 ноября 2025 года. Следовательно, CDSE нельзя считать рабочим fallback без отдельной правки provider-контракта и offline contract tests. OData должен оставаться резервным каналом поиска/скачивания, потому что CDSE прямо описывает STAC как дополнительный каталог с ограниченным набором коллекций.

### P0-3. Нет научного benchmark и калибровки

`confidence` рассчитывается как:

```text
0.50 * peak_score
+ 0.35 * clamp(contrast_sigma / 8)
+ 0.15 * clamp((elongation - 1) / 5)
```

Это полезный ranking score, но не вероятность. Нет размеченной выборки, negative-only сцен, false-alarm rate на площадь, stratified split и доверительных интервалов. Значения `3.5`, `31`, `5`, `min_area_px=2` сейчас являются эвристическими параметрами, а не результатом оптимизации на независимой выборке.

### P0-4. Wake wavelength пока не является валидной оценкой скорости

Текущий код строит **один cross-axis профиль возле кандидата**, ищет в нём повторяющиеся яркие пики и подставляет медианное расстояние в deep-water Kelvin formula. Этого недостаточно, чтобы утверждать, что измерена `Lmax` поперечной волны Kelvin wake:

- не доказано, что Hough-линия является wake axis, а не берегом/волновым фронтом/артефактом;
- не выделены Kelvin arms и не проверена принадлежность пиков к одной дисперсионной структуре;
- не учитываются конечная глубина, течение, wind-sea, угол визирования и ориентация пиксельной сетки;
- нет оценки неопределённости и минимальной длины наблюдаемого следа;
- для Sentinel-1 и Sentinel-2 не разделены разные механизмы формирования контраста.

Метод должен оставаться research-only: результат хранить как `speed_proxy_knots`, а не как оперативную скорость, пока не появятся paired AIS/разметка и QC.

## 5. Научные и алгоритмические слабые места

### 5.1. CFAR не является полноценным guard-cell CFAR

В `cfar_mask()` локальные `mean/std` вычисляются окном, которое включает саму цель. `guard_window_px` применяется после thresholding как `maximum_filter`, а не для исключения guard ring из оценки clutter. На ярких целях фон загрязняется целью, а на краях и рядом с NaN статистика меняет эффективное окно. Следующий detector должен явно разделять:

```text
training ring → guard cells → CUT (cell under test) → robust clutter model → threshold
```

Нужны как минимум valid-count, edge flag, robust quantiles/MAD либо распределение sea clutter и контроль false alarms per km².

### 5.2. Preprocessing sensor-specific не подключён

`config/processing.yaml` декларирует Lee filter, Sentinel-2 bands, SCL/cloud/water mask и tile parameters, но pipeline их не применяет. Сейчас:

- Sentinel-1 читает первый band выбранного raster asset и нормализует его глобальными percentiles;
- единицы продукта (amplitude/DN, sigma0, gamma0, dB) не фиксируются в provenance;
- Sentinel-2 также выбирает один band и пропускает cloud/water mask;
- tile/overlap из конфигурации не используются.

Пока это не исправлено, нельзя смешивать S1/S2 в одной шкале confidence или сравнивать результаты между провайдерами.

### 5.3. Wake association не защищён от ложных линейных объектов

Hough выполняется по исходному crop без land/shoreline mask, без suppression image border и без проверки протяжённости от корпуса в кормовом секторе. Линия принимается, если проходит в пределах 24 px от centroid. В результате берег, край AOI, радиальные волновые фронты и текстура моря могут получить `heading_method=wake_axis`.

Нужны: маска суши в том же crop, water-only valid mask, сектор за корпусом, минимальная длина/непрерывность, bilateral/arm symmetry, локальный contrast вдоль линии и сравнение с альтернативными line hypotheses.

### 5.4. AIS используется как сопоставление, но ещё не как строгая валидация

Текущее сопоставление интерполирует track point, но не ограничивает максимальный разрыв между соседними AIS сообщениями и не учитывает неопределённость времени/геометрии снимка. В pipeline нет полноценного `matched / unmatched / ambiguous / stale / out_of_window` QC. `validation.py` не вызывается, а `validation_status` остаётся `unvalidated`, кроме простого `ais_matched`.

AIS SOG/COG нужно хранить как reference с provenance, а не использовать для незаметной подмены собственного результата. Отдельно хранить расстояние, временной gap, heading difference и число track points.

### 5.5. Провайдеры смешивают search-only и detection-capable контракты

- ASF возвращает полезные Sentinel-1 metadata/preview и ZIP/GRD, но текущий materializer их не обрабатывает.
- Planetary Computer Sentinel-1 RTC практичен для COG, но официальный dataset отмечает необходимость аккаунта для SAS token; текущая документация ошибочно говорит, что credentials не требуются.
- CDSE STAC может отдавать COG bands, но требует нового endpoint/collection mapping и авторизации для защищённых assets.
- Sentinel Hub Catalog требует OAuth и не гарантирует, что Catalog item содержит прямой processable COG.

В интерфейсе нужно разделить `search_capable`, `preview_capable`, `raster_capable`, `archive_capable` и не считать сцену обнаруживаемой только по наличию metadata.

### 5.6. Геометрия AOI и память

Некоторые provider adapters берут только первый feature из FeatureCollection. Нет явного контракта для MultiPolygon/antimeridian. Большие raster assets читаются целиком; при отсутствии AOI crop это создаёт риск памяти. Raster cache не имеет lock-файлов, хотя отсутствие lock уже отмечено в плане.

## 6. Синтез признаков для будущей детекции

Признаки следует хранить раздельно по группам и не сводить сразу в произвольную одну confidence.

### 6.1. Vessel evidence

- local CNR/contrast, robust background scale, peak-to-clutter ratio;
- area, equivalent diameter, length/width, elongation, compactness, solidity, rectangularity;
- distance to coast/land mask, distance to AOI edge, valid-water fraction;
- incidence angle, orbit direction, instrument mode, polarization;
- dual-pol features `VV`, `VH`, `VV/VH` и cross-pol contrast, когда доступны;
- detector stability при изменении window/threshold и повторной обработке с overlap;
- saturation/nodata/edge flags.

### 6.2. Wake evidence

- normalized Hough/Radon accumulator per line length, continuity and gap fraction;
- distance from vessel candidate to wake onset and wake length;
- along-axis and cross-axis contrast, width growth and texture anisotropy;
- number of arms, arm-angle symmetry, vertex proximity, Kelvin-angle residual;
- positive and negative wake components, not only bright ridges;
- spectral/autocorrelation peak stability along the arm, not one profile at the vessel;
- agreement of wake axis with hull orientation when hull is resolvable.

### 6.3. Environment and physics

- wind speed/direction and sea-state proxy;
- current vector and wave height/period from Copernicus Marine or a documented fallback;
- bathymetry/depth and finite-depth flag;
- sensor incidence/azimuth and pixel anisotropy;
- daylight/cloud/glint/SCL flags for optical scenes.

Эти параметры дают не только новый score, но и объяснимый rejection: «кандидат есть, но условия наблюдения не позволяют подтверждать wake/speed».

### 6.4. Новая схема качества

Предлагается хранить четыре независимых результата:

```text
ship_score       — сила признаков корпуса/объекта
wake_score       — сила признаков следа
quality_score    — пригодность сцены и геометрии
uncertainty      — ошибка координат/курса/скорости и quality flags
```

До калибровки на разметке слово `confidence` в пользовательском интерфейсе заменить на `evidence_score` или `ranking_score`. `vessel_confirmed` выставлять только по порогу, зафиксированному на validation split.

## 7. Доступность данных и рекомендуемая иерархия

| Уровень | Источник | Реалистичное назначение |
|---|---|---|
| Primary discovery | CDSE STAC v1 | Sentinel-1 GRD / Sentinel-2 L2A metadata и COG assets; нужен новый endpoint и collection mapping |
| Primary processing fallback | Planetary Computer | S1 RTC/S1 GRD COG, S2 L2A; доступность SAS/auth проверять preflight-ом |
| Search/download fallback | CDSE OData | Поиск/загрузка полного продукта, когда STAC не видит сцену или asset |
| Search-only until processor | ASF | Metadata, preview, ZIP/GRD; включать в detection только после SAFE/GRD materializer |
| Optional commercial/API | Sentinel Hub | OAuth Catalog/Process; не считать бесплатным без отдельной проверки quota/contract |
| Ocean context | Copernicus Marine | Течение, волны, SST и ветер; подключать только с dataset id, units и temporal interpolation |
| Reference truth | Local AIS / NOAA MarineCadastre | Validation; NOAA — исторический американский охват, не глобальный realtime |

Запрещается выдавать preview за full-resolution asset и считать наличие STAC item гарантией доступности raster. Каждый запуск должен сохранять `asset_key`, media type, href scheme, CRS, GSD, band, units, processing level, auth mode и фактический download status.

## 8. Обновлённое ТЗ и границы MVP

### MVP-0.2: воспроизводимый single-raster candidate detector

Обязательный результат:

1. AOI и UTC interval обрабатываются с единым provider contract.
2. Выбирается только реально читаемый GeoTIFF/COG с описанным band/unit/CRS/GSD.
3. Выполняется sensor-specific preprocessing и water/land validity mask.
4. Результат называется `vessel_candidate`, пока нет benchmark confirmation.
5. Для кандидата сохраняются координаты, uncertainty proxy, evidence features и quality flags.
6. Wake axis выдаётся только при прохождении геометрического QC; иначе `heading_method=not_estimated`.
7. `speed_method=not_estimated` по умолчанию. AIS SOG — внешний reference. Kelvin wavelength — только research field с экспериментальным флагом и неопределённостью.
8. Обязательные артефакты: GeoJSON, CSV/Parquet, overview/crops, `report.json`. HTML переносится в отдельную задачу.
9. Результат воспроизводим по `product_id + asset_key + processing_config + code_commit`.

### MVP-0.3: научно валидированный S1 detector

Добавляется:

- правильный guard-cell CFAR/robust sea-clutter model;
- S1 calibration/unit contract и dual-pol features;
- coastline/edge/water masks;
- wake geometry с arm/continuity/angle QC;
- AIS matching quality and physical validation;
- curated benchmark и stratified evaluation;
- calibrated probability only after calibration split.

### Research-0.4

- Kelvin wavelength/spectral estimator с along-arm sampling, finite-depth/current correction и uncertainty;
- Sentinel-2 multi-band optical detector и validated inter-band timing/geometric model;
- ASF SAFE/GRD processing;
- multi-scene temporal association and track building;
- optional ML model only after a classical baseline and dataset card.

## 9. Критерии приёмки

### Engineering gate

- `pytest`, `ruff`, `bash -n`, clean install и core runtime check проходят;
- provider contract tests offline проверяют endpoint, collection, asset selection и понятный fallback;
- повторный запуск не создаёт повреждённых cache files и не скачивает один asset конкурентно дважды;
- все текущие output artifacts читаются и имеют provenance;
- сервис на ошибке доступа не выдаёт ложный «0 судов».

### Scientific gate

Разделить выборку по sensor/polarization/incidence/wind/depth/coast/open-sea/day-night/cloud и не смешивать соседние кадры одного прохода между train и test.

Метрики:

- detection: precision, recall, F1, POD, FAR, CSI, false alarms/km², median/p95 localization error;
- wake: wake detection rate, false-wake rate, angular MAE/p95 и arm/continuity QC;
- speed: bias, MAE, RMSE, median absolute error, coverage and uncertainty calibration against AIS/independent reference;
- operational: latency, bytes downloaded per request, cache-hit rate and provider success rate.

Порог научной приемки задаётся после pilot benchmark: новая версия должна превосходить простую CFAR baseline на независимом test split с bootstrap confidence intervals. Нельзя объявлять фиксированные проценты качества до сбора representative sample.

## 10. Приоритетный план после аудита

1. **P0 — вернуть release gate в зелёное состояние.** Обновить stale tests согласно текущему контракту, исправить два import-блока, сохранить тест на контраст детерминированного фона.
2. **P0 — восстановить data access.** Вынести `CDSE_STAC_URL`, `CDSE_STAC_SENTINEL1_COLLECTION`, `CDSE_STAC_SENTINEL2_COLLECTION`, добавить OData fallback и offline provider contract tests.
3. **P0 — зафиксировать честную семантику output.** Ввести candidate/evidence/quality/uncertainty; убрать operational speed из пользовательской выдачи до QC.
4. **P1 — подключить processing config.** Одинаково применять YAML/env к CLI и Telegram; писать полный effective config в report.
5. **P1 — переработать S1 preprocessing/CFAR.** Calibration/unit, valid mask, guard ring, robust clutter, tiled inference и overlap deduplication.
6. **P1 — переработать wake association.** Mask, background suppression, arm/sector/continuity tests, orientation and geometry QC.
7. **P1 — встроить физическую/AIS validation.** Gap/time uncertainty, current/wave context, explicit statuses and error metrics.
8. **P1 — собрать benchmark.** Scene manifest, labels, negative scenes, AIS provenance, data card, fixed split and reproducible evaluation CLI.
9. **P2 — Sentinel-2 full stack.** B02/B03/B04/B08, SCL/cloud/water mask, band timing and optical-specific detector.
10. **P2 — ASF processing и temporal tracking.** Только после стабильного S1 COG baseline.

До выполнения пунктов 1–3 новые providers, ML-модели и «улучшение» wake-speed не должны считаться приоритетом.
